# Developing enttui itself

Reference for contributors / AI agents modifying the enttui module. End users should start with the [README](../README.md).

## On-disk layout

```
enttui/
├── go.mod                              # module: enttui
├── README.md                           # user-facing entry doc
├── annotation.go                       # all schema.Annotation types
├── runtime/                            # handwritten generic tview app
│   ├── spec.go                         # EntitySpec[T], Column[T], EdgeSpec[T], ListOpts, Row
│   ├── registry.go                     # Register[T] + type-erased *anySpec
│   ├── app.go                          # App, page stack, global keys
│   ├── browser.go                      # one page: list + preview + status bar
│   ├── picker.go                       # kind picker modal
│   ├── preview.go                      # text/template renderers
│   ├── theme.go                        # tview.Styles overrides + tone colors
│   └── templates/
│       ├── preview.tmpl                # right-pane layout
│       └── status.tmpl                 # bottom status bar
├── gen/                                # codegen package
│   ├── generate.go                     # Generate(Options) entrypoint
│   └── templates/
│       ├── entity.tmpl                 # one per-entity register file
│       └── register_all.tmpl           # aggregator
├── cmd/enttui/                         # CLI wrapping gen.Generate
│   └── main.go
├── examples/
│   └── aicoder/                        # demo wiring this repo's schemas
│       ├── go.mod                      # separate submodule

│       └── gen/                        # generated, regenerable, committed
├── docs/                               # ← you are here

```

## Workflow

### Running the existing example

```bash
cd enttui && task run
```

Or by hand:

```bash
go run ./enttui/cmd/enttui --schema ./dbent/schema --out ./enttui/examples/aicoder/gen --package enttuigen
go run ./enttui/examples/aicoder --db /path/to/aicoder.db --project <id>
```

### Adding a new annotation

1. Declare the struct in `enttui/annotation.go`. Pattern:

   ```go
   type FooBar struct {
       schema.Annotation
       Value string
   }
   func (FooBar) Name() string { return "EntTUI.FooBar" }
   ```

2. Document it in [ANNOTATIONS.md](ANNOTATIONS.md).
3. (M1+) Read it inside `codegen/generate.go` from `field.Annotations` / `n.Annotations` and populate the relevant `EntityMeta` / `FieldMeta` / `EdgeMeta` field.
4. Reference the new metadata from the template.
5. Regenerate the example: `cd enttui && task gen` and confirm output diff.

### Adding a new field-Kind discriminator

If ent grows a new field type (rare):

1. Update `fieldKind(*gen.Field)` in `codegen/generate.go` to return a new bucket string.
2. Add a `{{- else if eq .Kind "newKind" }}` branch in the `valueExpr` template in `codegen/templates/entity.tmpl`.
3. Document the kind in [CONVENTIONS.md § Field-type kind discrimination](CONVENTIONS.md#field-type-kind-discrimination).

### Adding a new pane / page type to the runtime

See [RUNTIME.md § Adding a new pane / page type](RUNTIME.md#adding-a-new-pane--page-type) — concise checklist for wiring a new tview widget into the App + pages stack.

### Modifying templates

- **Runtime templates** (`runtime/templates/*.tmpl`) — only affect display. Hot-reload by stopping/starting the binary. No regen needed.
- **Gen templates** (`codegen/templates/*.tmpl`) — affect the **generated code**. After editing, regenerate (`task gen`) and check the diff in `examples/aicoder/gen/`. Then build the example to confirm Go compiles.

## Commit conventions

Match the parent repo (`type(scope): description`):

- `feat(enttui): add Sortable annotation`
- `fix(enttui): nil-guard optional time fields in codegen`
- `docs(enttui): expand CONVENTIONS reference`
- `refactor(enttui/runtime): pull preview rendering into preview.go`
- `chore(enttui): regen aicoder example after schema change`

## Style

- One file per concern in `runtime/`. Don't grow `browser.go` past ~400 lines — split if needed.
- Generated files must always start with `// Code generated by enttui. DO NOT EDIT.`.
- Templates: keep tview color tags inline with content. Don't pre-compute styled strings in Go and pass them to templates.
- Conservative on dependencies. Anything new goes in `examples/aicoder/` only; the runtime + gen packages keep their stdlib + tview + ent footprint.

## Known foot-guns

1. **`strings.Title`** is deprecated. Use `e.StructField()` (which gives the Go method name on the ent client) for edge method generation, not manual title-casing.
2. **`tview.ColorWhite`** is ANSI light-gray on many terminals. For true white use `tcell.NewRGBColor(255, 255, 255)` — or better, don't force a fg color at all, let the terminal default apply.
3. **`fmt.Sprintf("%v", any)`** can recurse infinitely on nested ent structs (Edges field contains other entities). The codegen never produces this — every accessor is typed. If you're tempted to use reflection-based dumping, **stop and add a typed accessor instead**.
4. **lipgloss `Width(w)` triggers `cellbuf.Wrap`** — even though we don't use lipgloss in v1, be aware if you swap renderers.
5. **Picker keystroke leakage** — single-letter shortcuts (`k`, `q`, `?`) must be suppressed while a tview.InputField is focused, or typing "task" into the picker reopens the picker. The check is `_, typing := a.tv.GetFocus().(*tview.InputField); if typing { ... }`.
